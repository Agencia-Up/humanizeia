## Resumo

A F2.6H estava implementada no working tree (não commitada). Claude AUDITOU contra os 6 requisitos,
achou a base sólida, aplicou **1 correção defensiva** (guarda seller × message_update), rodou todos os
gates reais e entrega para auditoria do Codex. Nada foi ativado, nenhum deploy, nenhum `db push`.

O F2.6H fecha o bloqueador do F2.6G: a Uazapi devolve `accepted` no send; a memória só avança com
`delivered`. Agora o callback `messages_update` promove o outbox por `providerMessageId` de forma
idempotente, sem reenviar mensagem.

## O que foi verificado/feito (6 requisitos)

1. **Endpoint `/v1/pilot/receipt`** (`Agent/src/runtime/pilot-http-app.ts`)
   - POST; **bearer igual ao turno** (`authorized()` com `timingSafeEqual` + digest privado, ANTES do roteamento — `handle()` linha 200).
   - Escopo: `isPedroV3PilotScope({tenantId, agentId})` → 403 `pilot_scope_denied` (só o par exato do piloto).
   - Só `delivered`/`read` (`parseReceiptPayload` rejeita o resto → 400); `providerMessageId` sem espaço e ≤240; `occurredAt` ISO válido.
   - Erros sanitizados (`receipt_payload_invalid`/`pilot_scope_denied`/`receipt_runner_unavailable`/`receipt_failed`); `catch` genérico; nenhum token/segredo.
   - Runner real wirado em `server.ts` (`ProductionPilotRunner` implementa `PilotReceiptRunner`; `new PilotHttpApp(secret, runtime, runtime)`) — não cai em 503 em produção.

2. **SQL manual** (`Brain/sql/v3_f2_6h_receipt_patch.sql`, incorporado em `v3_schema.sql`)
   - RPC `v3_find_outbox_by_provider_message_id(p_tenant_id, p_provider_message_id)`: `security definer`, `search_path=public`, valida entrada, filtra por **tenant_id** + `kind='send_message'` + `status='succeeded'` + `receipt_level in (accepted, delivered)`.
   - **Ambiguidade falha fechado**: `limit 2` no SQL + o adapter rejeita quando `data.length !== 1`.
   - `revoke all from public; grant execute to service_role`. Índice parcial por (tenant, providerMessageId).
   - Cabeçalho: "Owner applies this patch manually in Supabase SQL Editor. Never via db push."

3. **Persistência** (`Agent/src/adapters/persistence/postgres-store.ts`, `Agent/src/engine/provider-delivery-receipt.ts`)
   - `findOutboxByProviderMessageId` via **RPC allowlist** (gateway permite só `v3_find_outbox_by_provider_message_id` e companhia); valida id (1..240); ambiguidade → throw.
   - `applyProviderDeliveryReceipt`: `not_found` sem erro; `kind` validado; **idempotente** — `receiptLevel==='delivered' && outcomeAppliedAt!==null` → `duplicate` (não recommita, não reenvia); senão promove accepted→`delivered` via `commitEffectOutcome` (CAS, terminal não reabre).
   - `accepted` puro NÃO avança memória como entregue; `delivered`/`read` avança idempotente.

4. **Uazapi** (`Agent/src/adapters/effects/uazapi-whatsapp-sender.ts`)
   - Captura o `messageid` oficial (loop `messageid/messageId/message_id/id/key`).
   - Inclui `track_source: "pedro_v3"` + `track_id: idempotencyKey` nos bodies suportados (correlação do receipt).
   - Não reenvia em receipt duplicado (garantido pela guarda `duplicate` do engine).
   - **Config de webhook (messages_update): ver "Manual do dono" abaixo** — o path moderno (`create-uazapi-instance` `/webhook/instance`) já inclui `messages_update`; não foi alterada config compartilhada do v2 (isolamento).

5. **Bridge v2** (`supabase/functions/_shared/pedro-v2/pedroV3Bridge.ts`, `supabase/functions/pedro-webhook-v2/index.ts`)
   - `isMessageUpdateEvent` cobre `messages_update`/`message_update`/`messages.update`; interceptado **ANTES do ignore por `fromMe`**.
   - `buildPedroV3DeliveryReceipt`: identidade **hardcoded** nas constantes do piloto (não do payload); não-piloto → `not_pilot_identity`; status fora de delivered/read → `status_ignored`; `providerMessageId` validado.
   - Piloto+`active` → `callPedroV3ReceiptBridge` via `EdgeRuntime.waitUntil` (não bloqueia, não reenvia); demais → `ignored: "message_update"`. **Nunca inicia conversa nova.**
   - `callPedroV3ReceiptBridge`: valida secret (≥32), timeout+abort, `redirect:"error"` (anti-SSRF), limite de bytes, Bearer, sem token no retorno, `catch`→`uncertain` (nunca alega entrega em erro).
   - **Correção defensiva aplicada**: guarda no ramo do vendedor (`seller_member_id`) para ignorar `message_update` e nunca poluir o inbox do vendedor com entrada vazia (sem quebrar v2).

6. **Build** — `PEDRO_V2_BUILD` `2026-06-28-pedro-v3-active-bridge-v220` → `2026-06-28-pedro-v3-delivery-receipt-v221`. Bundle do webhook OK.
