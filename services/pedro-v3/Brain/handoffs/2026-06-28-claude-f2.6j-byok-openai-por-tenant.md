`PEDRO_V3_PILOT_MODE` continua OFF.

## Bloqueador (achado pelo Codex)

`server.ts` exigia `OPENAI_API_KEY` global via env. Isso fere o produto: a chave de IA deve vir do
perfil/portal do tenant (BYOK), como o Pedro v2 ja faz em `_shared/aiKeys.ts`
(`resolveAiKey` -> RPC `get_client_ai_key`, Vault, service-role).

## O que foi feito (regras F2.6J)

1. **Sem chave global no servico**: removido `requiredEnv("OPENAI_API_KEY")` e o campo `#openAiKey` de
   `ProductionPilotRunner`. O servico nao le mais essa env (provado por teste estrutural).
2. **Resolucao por tenant**: novo `Agent/src/adapters/read/tenant-openai-key.ts` ->
   `resolveTenantOpenAiSecret({ gateway, tenantId })` chama a **mesma RPC do v2** `get_client_ai_key`
   (`p_user_id = payload.tenantId`, `p_provider = "openai"`) pela `SupabaseServiceGateway` (service-role).
   `get_client_ai_key` entrou no `RPC_ALLOWLIST` do gateway.
3. **Sem fallback global/silencioso**: a chave da plataforma NAO e usada. Sem chave valida do tenant,
   `resolveTenantOpenAiSecret` lanca; no `run()` isso cai no catch -> `PILOT_BOOTSTRAP_FAILED` com
   `ingested=false` (sem dispatch, sem mensagem dupla; o contrato anti-resposta-dupla decide o fallback v2).
4. **Falha fechada + erro sanitizado**: chave ausente/nula/com-espaco -> `OPENAI_KEY_NOT_FOUND`;
   erro de leitura do gateway -> `OPENAI_KEY_LOOKUP_FAILED` (NUNCA propaga o erro original/corpo/segredo);
   `tenantId` vazio -> `TENANT_INVALID` (nem chama o gateway).
5. **Sem vazamento**: a chave volta SO embrulhada em `OpenAiRuntimeSecret` (opaca) — `#apiKey` privado,
   `toJSON()` -> `{kind:"openai_runtime_secret"}`, liberada apenas via `materialize()` no header do adapter.
   Nada de chave em log, `JSON.stringify`, erro, estado, evento, outbox ou QueryResult (testado).
6. **Modelo inalterado**: `PEDRO_V3_OPENAI_MODEL=gpt-4.1-mini` segue como default/override (so o NOME do
   modelo); a CREDENCIAL vem do tenant.
7. **Docs**: `README.md` removeu `OPENAI_API_KEY` das obrigatorias + nota BYOK; handoff F2.6I corrigido.
