## Cadeia de diagnostico (a observabilidade pagou)

1. F2.6L/M -> `v3_inbox.last_error` passou a gravar o motivo do turno (antes invisivel: o engine falha gracioso).
2. F2.6N -> "claimed inbox record missing" era DOUBLE-ENCODING no filtro PostgREST. Corrigido. Avancou 1 camada.

4. Log do Postgres (get_logs): **`ERROR: new row for relation "v3_turn_events" violates check constraint
   "v3_turn_events_payload_ck"`**.
5. Provado no banco (regex isolada): o payload do evento `turn_claimed` inclui os `event_ids` do uazapi (hash
   hex de 64 chars); um hash com **11 digitos seguidos** (`77842555836` dentro de `...f77842555836c...`) batia
   no heuristico de CPF e o check rejeitava -> 23514 -> PostgREST 400.

## Causa raiz

`v3_payload_is_redacted` (usada pelos checks `v3_turn_events_payload_ck` e `v3_outbox_payload_ck`) detectava CPF
com `'(^|[^0-9])[0-9]{3}[.]?[0-9]{3}[.]?[0-9]{3}-?[0-9]{2}([^0-9]|$)'`. As bordas `[^0-9]` tratam letras hex
(a-f) como delimitador valido, entao 11 digitos GRUDADOS dentro de um hash viravam falso positivo. Nunca rodou
contra o Postgres real (o `get()` quebrava antes do commit); os testes usavam ids sem run de 11 digitos.

## Fix

Trocar as bordas por **word-boundary `\y`**: `'\y[0-9]{3}[.]?[0-9]{3}[.]?[0-9]{3}-?[0-9]{2}\y'`. Sequencias de
digitos grudadas em alfanumerico (hashes) nao casam; CPF real (formatado OU cru, cercado por espaco/pontuacao/
inicio/fim) continua pego. Provado no banco e no teste:
- `..f77842555836c..` -> regex nova = FALSE (passa) ; regex antiga = TRUE (barrava).
- `123.456.789-00` e ` 12345678900 ` -> regex nova = TRUE (barrado, como deve).

`create or replace function` preserva grants; os checks passam a usar a versao nova nos proximos inserts.
