## Contexto

Apos F2.6Q o Pedro v3 responde de PONTA A PONTA no WhatsApp (recebe->decide->commita->ENVIA->chega no celular).
Mas toda decisao gravava `reason_code=error`, `reason_summary="Erro/Timeout no passo propose: MODEL_DECISION_INVALID"`
e o lead recebia sempre a MESMA frase de fallback ("Desculpe a lentidao temporaria...") = loop.

## Causa raiz

O passo `propose` falha em `decodeProposal`/`decodeStep` (prompt-bound-conversation.ts) -> `emitErrorTerminalSafe`
(finalizer.ts:309) gera um `send_message` "safe-terminal" generico. O motivo: a instrucao enviada ao modelo
(`operationInstructions("propose")` em openai-chat-model.ts) dizia apenas *"Return one JSON object matching
DecisionStep / ProposedDecision"* mas **NUNCA descrevia essas estruturas** (campos `proposedAction`, `facts`,
`proposedEffects`, `responsePlan.guidance`, `reasonCode`, `confidence`, e o shape de DecisionStep query|final).
O modelo nao tem como produzir um envelope que nunca lhe foi mostrado -> JSON valido porem fora do contrato ->
rejeitado. Passou despercebido pq os testes usam um modelo FAKE que ja devolve o envelope certo (nunca exercitou
um modelo REAL contra a instrucao).

Achado critico no finalizer: no caminho normal `effectPlan = proposal.proposedEffects` (linha 200-202) — **nao ha
auto-criacao de send_message**. Logo, para enviar uma resposta, o modelo PRECISA emitir um efeito send_message.

## Fix

1. `openai-chat-model.ts` -> `operationInstructions` reescrito para os 3 passos, descrevendo o envelope EXATO
   (igual aos validadores) com exemplos concretos:
   - interpret: `{relation: <enum>, intentSummary?, extractedEntities?}`.
   - propose: DecisionStep = `{kind:"query",call:{tool,input}}` (4 tools + inputs) OU `{kind:"final",proposal:{...}}`
     com TODOS os campos + valores permitidos de proposedAction. **Regra destacada**: para responder, incluir
     `{"kind":"send_message","planId":"reply","order":0,"onSuccess":[]}` em proposedEffects (o TEXTO vem do compose,
     guiado por responsePlan.guidance). Conservador: instrui `facts: []` (sem mutacoes de estado) para nao arriscar
     mutacoes invalidas neste 1o corte — slot/objetivo fica para a proxima iteracao.
   - compose: ResponseDraft = `{parts:[...]}` com os 3 tipos de part (text/vehicle_ref/money_ref) + exemplo.
2. `prompt-bound-conversation.ts` -> `ModelOutputError` ganha `detail` opcional; cada checagem do decoder agora
   nomeia o campo que falhou (ex.: `MODEL_DECISION_INVALID:proposedEffects`). So nome de campo, nunca dado do
   modelo. Aparece em `v3_decisions.reason_summary` -> diagnostico/iteracao rapida pelo banco.
