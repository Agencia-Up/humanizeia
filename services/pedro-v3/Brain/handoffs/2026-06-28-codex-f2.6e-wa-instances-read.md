# Handoff — F2.6E — wa_instances read-side seguro para piloto

Data: 2026-06-28
Executor/auditor: Codex
Escopo: Pedro v3 `Agent/` + `Brain/`; sem deploy, sem banco remoto, sem webhook ativo.

## Objetivo

Preparar a ponte segura entre o runtime ativo do piloto e a instancia WhatsApp existente no v2, sem ainda ligar o Pedro v3 para responder no WhatsApp real.

A premissa desta fatia foi: antes de qualquer envio ativo, o v3 precisa conseguir carregar a instancia do agente por `tenantId + agentId + instanceId`, validar ownership, separar metadata de segredo e resolver o token Uazapi somente no ponto de envio.

## Arquivos alterados

- `Agent/src/adapters/read/supabase-v2-read-adapter.ts`
  - adiciona a tabela `wa_instances` e colunas necessarias (`instance_name`, `api_url`, `provider`, `api_key`).
  - ajusta o type guard de stock provider para nao confundir `uazapi` com feeds de estoque.

- `Agent/src/adapters/read/supabase-read-database.ts`
  - adiciona capabilities estritas para `wa_instances/selectOne` metadata.
  - adiciona capability separada para `wa_instances/selectOne` de segredo.
  - segredo (`api_key_encrypted`/`api_key`) segue proibido em `selectMany`, sem `id`, sem `user_id`, ou fora da matriz.

- `Agent/src/adapters/read/v2-api-key-reader.ts`
  - passa a entender token Uazapi em formato JSON (`api_key`, `api_key_encrypted`, `token`, `apikey`) ou escalar cru, coerente com o achado factual do v2.

- `Agent/src/adapters/effects/pilot-whatsapp-runtime.ts`
  - `WhatsAppInstanceProvider` agora e `"uazapi" | "unsupported"`, removendo cast falso.

- `Agent/src/adapters/effects/v2-whatsapp-instance-source.ts`
  - novo source read-only de instancia WhatsApp.
  - valida `id`, `user_id`, `api_url` e provider.
  - provider vazio/null e tratado como Uazapi por compatibilidade com o v2.
  - provider diferente vira `unsupported` e o runtime bloqueia.
  - novo `V2WhatsAppInstanceCredentialProvider` resolve token via `SecretRef` e valida ownership/provider.

- `Agent/tests/run-active-effects.ts`
  - testes do source real de `wa_instances`, credential provider, cross-tenant e provider unsupported.

- `Agent/tests/run-canary-wiring.ts`
  - testes da matriz Supabase para `wa_instances`: metadata sem token, token apenas por `selectOne(id+tenant)`, bloqueio em lote/sem id/sem tenant.

## Invariantes provados

1. Metadata da instancia nao seleciona token.
2. Token de `wa_instances` nao pode ser lido em lote.
3. Token de `wa_instances` nao pode ser lido sem `id`.
4. Token de `wa_instances` nao pode ser lido sem `user_id`.
5. Cross-tenant nao retorna instancia nem segredo.
6. Provider nao-Uazapi e bloqueado antes do envio.
7. Token Uazapi so e resolvido no ponto de envio, por `SecretRef` opaco.
8. Nao ha cast `as never` para furar o tipo de provider.
