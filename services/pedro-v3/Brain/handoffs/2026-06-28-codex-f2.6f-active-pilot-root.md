# Handoff — F2.6F — active pilot root local

Data: 2026-06-28
Executor/auditor: Codex
Escopo: Pedro v3 `Agent/` + `Brain/`; sem deploy, sem webhook vivo alterado nesta fatia.

## Objetivo

Criar a camada ativa local que faltava entre o motor conversacional e o envio real de WhatsApp: um root unico que valida o piloto, ingere inbound, roda a decisao, persiste outbox e despacha WhatsApp via Uazapi somente quando o modo ativo esta permitido.

## Entrega

Arquivo novo:

- `Agent/src/engine/pilot-active-root.ts`

Ele compoe:

- `evaluatePedroV3PilotScope` para bloquear tudo que nao seja o tenant/agente piloto.
- `V2TenantConfigSource` para carregar prompt/config do portal.
- `V2StockLoader`, `V2StockSource`, `V2VehiclePhotoSource` e `V2CrmReadSource` para read-side.
- `PromptBoundConversationAdapter` + `ConversationTurnContextPreparer` para interpret/propose/compose state-aware.
- `ConversationEngine` para commit atomico de decisao/outbox.
- `createPilotWhatsAppDispatcher` + `OutboxDispatcher` para envio ativo de WhatsApp.

Arquivo novo de teste:

- `Agent/tests/run-active-root.ts`

Package:

- script `test:active-root`
- `test:all` passa a incluir a suite nova.

## Invariantes provados

1. Root ativo nao nasce para agente fora do piloto.
2. Root ativo falha fechado se o agente piloto nao tiver `instance_id`.
3. Um turno ativo valido faz commit e despacha exatamente uma mensagem WhatsApp.
4. Uazapi `accepted` nao e tratado como `delivered`; nao aplica outcome conversacional.
5. A memoria registra o inbound do lead, mas nao inventa que a resposta do agente foi entregue quando so houve `accepted`.
6. O prompt do portal chega ao modelo estruturado.
7. Mesmo `eventId` duplicado nao reprocessa nem reenvia.
8. `handoff` ainda nao executa provider externo nesta fase; cai em resposta WhatsApp segura em vez de transferencia silenciosa.
