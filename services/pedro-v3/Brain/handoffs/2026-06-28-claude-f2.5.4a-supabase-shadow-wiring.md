  - `CanaryShadowRoot` — composition root: tenant/agente **explícitos**, `mode` deve ser `"shadow"` (senão aborta), aborta se `EffectGate` injetado estiver ativo, monta o stack read-only real (gateway+credential provider+sources+QueryRunner) e roda via `runShadowHarnessTurn` (EffectGate OFF, recorder que falha se houver dispatch). Defesa final: lança se `dispatchAttempts != 0`. Sem dispatcher externo registrado. `assertCanaryGateShadow` exportado.
- `Agent/src/adapters/read/...` — **nenhum** arquivo da F2.5.3 foi alterado; o wrapper pluga atrás do contrato `V2ReadDatabase`/`SecretDecryptor` já existente.

- `Agent/tests/run-canary-wiring.ts` + `package.json` (`test:canary` + incluído no `test:all`).

## 2. O que foi apenas INSPECIONADO (read-only, v2)

- **Criação do client Supabase / leitura de tabelas:** o v2 usa o SDK em edge functions; o v3 NÃO copia o SDK — implementa um wrapper PostgREST próprio atrás do contrato.
- **Formato de `api_key_encrypted` (crítico):** comprovado **plaintext** (`parseCredentials` = `JSON.parse`/raw; `mediaContext`/`metaSender` usam o valor direto como token). → ADR-008. **Não inventei decryptor criptográfico.**
- **Hosts vivos** (já na allowlist do `http-client.ts` da fatia B): RevendaMais `app.revendamais.com.br`, BNDV `api-estoque.azurewebsites.net`.
- **Env/segredos:** nenhuma variável real foi lida; a chave Supabase é injeção de config (em teste é sintética `test-anon-key-NOT-REAL`).

## 3. O que foi realmente TESTADO (`run-canary-wiring.ts`, 33 checks)

- Config: HTTPS obrigatório, host fora da allowlist rejeitado, chave ausente fail-closed, **chave não serializável** (`JSON.stringify`/`toJSON`).
- Wrapper: select permitido; **tabela não permitida rejeitada**; **coluna não permitida rejeitada**; **ausência de filtro de tenant falha**; **escrita impossível** (sem insert/update/delete/upsert/rpc); **cross-tenant falha fechado** (via config source + db real → `AGENT_NOT_FOUND`); **erro do Supabase sanitizado** (exceção e status 500 não vazam URL/token).
- Decryptor: sintético válido (RevendaMais feed_url / BNDV api_token / raw); **payload adulterado/sem campo → null**; **payload vazio → null**; provider desconhecido → null.
- Canary root: aborta com `mode != shadow`; aborta sem tenant/agente; **não inicia com EffectGate ativo**; `assertCanaryGateShadow` aceita shadow / rejeita ativo; `runQuery` `crm_read` lê do **db real** (read-only); turno shadow commita, **0 dispatch**, outbox `skipped` sem `outcomeAppliedAt`, comparação passa.

## 4. O que permanece BLOQUEADO

- EffectGate ativo, providers reais, WhatsApp/CRM-write/handoff/agenda — todos OFF.

## 5. Riscos encontrados

- **Segredos de integração em repouso são PLAINTEXT** no `platform_integrations` do v2 (ADR-008). Recomendação: criptografia real em repouso numa fase futura (substituível atrás do mesmo `SecretDecryptor`).
- **`service_role` exposta** ainda pendente de rotação (pré-requisito do canary remoto).
- Wrapper Supabase: a proteção anti-SSRF do host é por **allowlist de host na config** (o host do Supabase é fixo/confiável, não entrada do usuário); não há resolução DNS anti-rebind como no `http-client` de estoque — aceitável pois o host não é controlável pelo LLM, mas registrado.
